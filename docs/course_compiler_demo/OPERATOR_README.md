# AxiomIQ Course Compiler Demo Operator README

## Purpose

The Course Compiler Demo is a non-live local prototype that converts a sample Algebra I source document into demo curriculum intelligence outputs. It produces source interpretation, topic candidates, micro-skill candidates, practice package previews, assessment package previews, performance tracking previews, content gap reporting, validation reporting, and a human-readable demo report.

This is not canonical curriculum.
This is not production-ready content.
This does not update operational Alpha readiness or mastery.
No database contact is required.
All generated learning outputs are demo_unverified.

## Non-Live Boundary

The demo runs entirely from local files in this repository. It does not contact a production database, hosted database, local database, student data store, live API, operational Alpha procedure files, operational Alpha question files, readiness files, mastery files, grading files, or routing files.

All learning outputs remain `demo_unverified`. That label means the output was generated by the local demo pipeline and has not been reviewed, approved, canonicalized, or made ready for live instruction.

## Run the Demo

From the repository root:

```bash
python3 tools/course_compiler_demo/run_course_compiler_demo.py \
  --input tools/course_compiler_demo/sample_inputs/math_demo_input.txt \
  --subject MATHEMATICS \
  --mode student_practice \
  --output reports/course_compiler_demo/latest
```

The final published MVP outputs are also frozen under:

```text
reports/course_compiler_demo/final_math_demo_001/
```

## Run the Smoke Test

From the repository root:

```bash
python3 tools/course_compiler_demo/smoke_test_course_compiler_demo.py
```

The smoke test runs the compiler against the sample math input and writes a fresh verification run to:

```text
reports/course_compiler_demo/smoke_test_run/
```

It exits `0` when the current MVP output shape passes the expected-output checks. It exits nonzero if required files, labels, counts, report markers, validation results, or forbidden-zone checks fail.

## Output Files

The demo output folder contains these files:

- `source_document.json`: local metadata for the source document, rights status, privacy status, and source identity.
- `source_interpretation.json`: detected source type, subject, course level, primary use, feature flags, and processing recommendation.
- `source_feature_flags.json`: structured feature signals detected from the source.
- `curriculum_extraction_package.json`: combined extraction package containing topic candidates, micro-skill candidates, targets, and gaps.
- `topic_candidates.json`: extracted topic candidates with evidence and demo alignment status.
- `micro_skill_candidates.json`: extracted micro-skill candidates with parent topic links and evidence references.
- `practice_targets.json`: demo practice-generation targets derived from extracted skills.
- `assessment_targets.json`: demo assessment targets derived from extracted skills.
- `performance_tracking_targets.json`: demo tracking targets for previewing readiness-style metrics.
- `content_gaps.json`: machine-readable demo content gap findings.
- `practice_module_package.json`: demo practice module package with source-derived practice items.
- `practice_assessment_package.json`: demo checkpoint assessment package with source-derived assessment items.
- `performance_tracking_package.json`: demo-only performance tracking preview package.
- `content_gap_report.md`: human-readable explanation of detected content gaps and recommended follow-up.
- `demo_report.md`: human-readable end-to-end summary of the demo run.
- `validation_report.json`: machine-readable validation summary for required files, status labels, boundary checks, and overall result.

## Interpreting validation_report.json

`validation_report.json` is the fastest machine-readable health check. A passing MVP run should have:

- `status` set to `demo_unverified`.
- `validation_status` set to `completed`.
- `overall_result` set to `pass` or `pass_with_warnings`.
- Empty `missing_files` for the expected MVP output set.
- Empty `required_field_errors`.
- `demo_label_check.passed` set to `true`.
- `db_contact_check.db_contact_performed` set to `false`.
- `forbidden_zone_check.backend_app_touched` set to `false`.
- `forbidden_zone_check.frontend_src_touched` set to `false`.
- `forbidden_zone_check.operational_alpha_touched` set to `false`.

Warnings should be read before sharing outputs, but `pass_with_warnings` is still an accepted demo validation state.

## Interpreting demo_report.md

`demo_report.md` is the operator-facing walkthrough of the run. It summarizes the input, source interpretation, topic map, micro-skill map, practice module, assessment package, performance tracking preview, content gaps, rights status, limitations, validation summary, and non-live boundary.

The report must preserve the `demo_unverified` and `Non-Live` boundary language. Any report that implies production readiness, canonical curriculum, live readiness updates, or mastery updates should be treated as invalid for this MVP.

## Interpreting content_gap_report.md

`content_gap_report.md` lists known gaps that limit instructional reliability and live-use readiness. These gaps are expected in the MVP because the demo is source-derived, local, and unreviewed. The report is not a blocker for demo execution, but it is a guide for future review, enrichment, canonical alignment, and production-hardening work.

## What the MVP Does Not Do

- It does not create canonical curriculum.
- It does not create production-ready lessons, procedures, questions, assessments, or mastery rules.
- It does not update operational Alpha readiness or mastery.
- It does not write to backend, frontend, migration, deployment, CI, grading, routing, readiness, mastery, or operational Alpha files.
- It does not contact any database, student data store, hosted service, or live API.
- It does not support broad subject coverage beyond the current math demo lane.
- It does not perform human review, final instructional approval, or production validation.

## Known Limitations

- Extraction is rule-based and intended for a narrow math demo input.
- Demo canonical matching uses the local demo registry only.
- Practice and assessment items are source-derived previews, not reviewed item-bank content.
- Performance tracking is a local preview and does not represent live learner state.
- Content gaps are expected because reviewed procedures and reviewed question banks are outside this MVP.
